The component responsible for outputting column values in grids is:

The cell rendering logic is defined at **lines 124-147**:

```javascript
cell: ({ getValue }) => {
  const value = getValue();
  // Handle null/undefined values
  if (value === null || value === undefined) return '-';  // <-- THIS IS THE ISSUE
  // Handle objects and arrays
  if (typeof value === 'object') return JSON.stringify(value);

  const stringValue = String(value);

  // Get field config for additional context
  const fieldConfig = collection?.fields?.[colDef.field];
  const isLongTextField = fieldConfig && ['textarea', 'markdown', 'wysiwyg'].includes(fieldConfig.type) ||
                          ['description', 'content', 'body', 'text', 'message', 'notes'].includes(colDef.field.toLowerCase());

  if (isLongTextField && stringValue.length > 100) {
    return (
      <span title={stringValue} className="cursor-help">
        {stringValue}
      </span>
    );
  }

  return stringValue;
},
```

## Issues

### ISSUE #1: Relation Field Values Display as "-" in Grid

**Severity:** High
**Location:** `react/packages/grid/src/App.js:124-147`

**Description:**
When a collection has a relation field (e.g., Tickets → TicketStatuses), the grid displays "-" instead of the selected value. This occurs because:

1. The RelationField component (`react/packages/fields/src/components/field-types/RelationField.js`) stores only the ID value (line 91: `value={option[valueField]}`)
2. The grid's cell renderer receives this ID value
3. However, the data returned from the API might not be resolving the relation, or the ID is not being properly expanded to show the label

**Root Cause Analysis:**
The grid cell renderer at line 127 checks `if (value === null || value === undefined) return '-'`. However, for relation fields, we may be getting:
- Just an ID (numeric value like `5`) which should be resolved to show the label
- An object that should be unpacked to show a specific property
- Or possibly the value is actually null/undefined when it shouldn't be

**Expected Behavior:**
Relation fields should display the related record's label field (e.g., "Open", "Closed") instead of "-" or the raw ID.

**Potential Solutions:**
1. Check if the API is returning expanded relation data or just IDs
2. Modify the grid cell renderer to handle relation field types specifically
3. Add a custom cell renderer for relation fields that looks up the label from field config
4. Ensure the collection data includes resolved relations (join/expand in backend)

**Next Steps:**
- [ ] Inspect actual API response data structure for Tickets collection
- [ ] Check if TicketStatuses field config includes relation metadata
- [ ] Determine if backend is expanding relations or returning raw IDs
- [ ] Implement appropriate cell renderer for relation field type

### ISSUE #3: Grid Columns Generated from Data Rows (Unreliable) ✅ FIXED

**Severity:** Medium (data integrity/predictability)
**Location:** `react/packages/grid/src/Grid.js:111-252`
**Status:** RESOLVED

**Description:**
The grid had a third fallback option that would generate columns from the first data record. This was unreliable because:
- Column order would be arbitrary based on object key order
- No field type information available
- No labels, just raw field names
- Inconsistent behavior if first record had different structure than others

**Previous Logic (3 priorities):**
1. Use `grid.columns` if defined
2. Use `collection.fields` (first 5)
3. **Generate from first data row (REMOVED)** - Lines 179-216

**Resolution:**
✅ Removed Priority 3 entirely
✅ Updated comment from "Generate columns from collection grid config or data" to "Generate columns from collection grid config or fields"
✅ If neither Priority 1 nor 2 exists, grid will show no columns (intentional - forces proper collection configuration)

**New Behavior:**
Grids now require either:
- Explicit `grid.columns` configuration in collection metadata, OR
- Collection `fields` defined (shows first 5 automatically)

This ensures predictable, consistent grid displays with proper labels and field type awareness.

### ISSUE #4: Column Generation Logic Mixed with Component Logic ✅ FIXED

**Severity:** Low (code organization/maintainability)
**Location:** `react/packages/grid/src/Grid.js`
**Status:** RESOLVED

**Description:**
The Grid component had ~70 lines of column generation logic embedded directly in a useMemo hook. This made the component harder to understand and maintain, as column generation is not actually part of the component's core responsibility.

**Resolution:**
✅ Created `react/packages/grid/src/services/columnGenerator.js` with `generateColumns()` utility
✅ Moved all column generation logic (Priority 1 & 2) into the utility
✅ Updated Grid.js to import and use `generateColumns(collection)`
✅ Reduced Grid.js from ~315 lines to 215 lines (~100 lines removed)

**New Structure:**
```javascript
// Grid.js - Now clean and focused
const columns = useMemo(() => {
  if (!data || data.length === 0) return [];

  // Get base columns from collection metadata
  const baseColumns = generateColumns(collection);

  // Add actions column if enabled (component-specific logic)
  if (showActions && (onEdit || onDelete)) {
    baseColumns.push({ /* actions column */ });
  }

  return baseColumns;
}, [data, collection, showActions, onEdit, onDelete]);
```

**Benefits:**
- Separation of concerns: column generation is now a pure utility function
- Grid component focuses on rendering and interaction logic
- Column generation logic can be easily tested in isolation
- Can be reused by other components if needed
- Much easier to understand and maintain

### ISSUE #5: Inconsistent Package Naming Between Development and NPM ✅ FIXED

**Severity:** High (build-breaking)
**Location:** Multiple app and package package.json files, and import statements
**Status:** RESOLVED

**Description:**
The project had mixed use of NPM published names (`@arcwp/gateway-*`) and local workspace aliases (`@gateway/*`), causing build failures:
- Apps used version numbers (`@gateway/forms: ^1.1.1`) instead of `file:` references
- Packages referenced NPM names (`@arcwp/gateway-filters`) instead of workspace aliases
- Import statements used NPM names instead of workspace aliases
- Workspace requires consistent use of `@gateway/*` with `file:` references

**Files Fixed:**

**Apps (use `@gateway/*` with relative file paths):**
- ✅ `apps/studio/package.json` - Fixed forms & grid references
- ✅ `apps/admin/package.json` - Fixed forms & grid references
- ✅ `apps/form/package.json` - Fixed forms reference
- ✓ `apps/grid/package.json` - Already correct
- ✓ `apps/filters/package.json` - Already correct

**Packages (use `@gateway/*` for inter-package dependencies):**
- ✅ `packages/grid/package.json` - Changed `@arcwp/gateway-filters` → `@gateway/filters`
- ✅ `packages/forms/package.json` - Changed `@arcwp/gateway-fields` → `@gateway/fields`

**Import Statements:**
- ✅ `packages/forms/src/index.js` - Changed `@arcwp/gateway-fields` → `@gateway/fields`
- ✅ `packages/forms/src/components/FormBuilder.js` - Changed `@arcwp/gateway-fields` → `@gateway/fields`

**Resolution Pattern:**
```json
// In apps and packages - use @gateway/* with file: references
{
  "dependencies": {
    "@gateway/forms": "file:../../packages/forms",
    "@gateway/grid": "file:../../packages/grid",
    "@gateway/fields": "file:../fields",
    "@gateway/filters": "file:../filters"
  }
}
```

**Naming Convention:**
- **Development (workspace):** Use `@gateway/*` with `file:` references
- **NPM Publishing:** Packages publish as `@arcwp/gateway-*`
- **Imports:** Always use `@gateway/*` in source code
